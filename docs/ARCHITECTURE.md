# Architecture Overview

## Core Principle

Strategic Nexus must not do anything illegal or against OpenAI rules.

If a proposed architecture path could violate law, third-party rights, platform terms, OpenAI policy, or safe-use boundaries, it is out of scope.

The external AI layer NEVER directly controls the game.

It only:
- analyzes
- interprets
- recommends
- modifies strategic priorities

The safe architecture has three layers:

1. Vanilla Stellaris mod
2. External daemon
3. File bridge / scripted receiver artifacts

The vanilla mod owns all in-game behavior.
The daemon is a passive external intelligence worker.
The file bridge validates bounded data and never modifies the running game process.

The daemon must not aggressively poll the game or scan game state every frame.
It waits for explicit strategic work generated by an exchange source or by an offline MVP workflow.
The mod must not be responsible for launching the daemon process.
The design must not assume the vanilla mod can read arbitrary external files.

The daemon has its own lifecycle as an independent background application:

```text
Windows/user starts StrategicNexusDaemon
    -> daemon idles
    -> mod writes request file
    -> daemon wakes on file change
    -> daemon writes response file
    -> daemon returns to sleep
```

This avoids launch permissions, antivirus friction, sandbox assumptions, and mod API hacks.

Correct flow:

```text
Vanilla Stellaris Mod
    -> runs scripted fallback behavior by default
    -> optionally receives small scripted state values through safe development artifacts

External Daemon
    -> idles as an independent process
    -> analyzes explicit request/snapshot data
    -> writes strategic response for bridge/diagnostics

File Bridge / Scripted Receiver
    -> validates daemon output if available
    -> maps only bounded values to allowlisted artifacts
    -> fails closed to vanilla scripted fallback
```

Incorrect flow:

```text
daemon continuously scans runtime game state
daemon blocks gameplay while waiting for inference
daemon modifies saves or running game internals directly
daemon replaces vanilla AI
bridge exposes raw AI output or tactical commands
mod depends on external file reads
```

The game remains the master.
The daemon remains advisory.
The mod never waits synchronously for daemon output.
If the bridge fails, the mod continues with fallback scripted behavior.

No AI replacement.
No save editing.
No direct tactical control.
No fleet control.
No arbitrary command strings from daemon.
No raw LLM text enters the game.

Ironman compatibility is not required for v0.x.
Do not optimize v0.x for achievements.
Non-Ironman runtime integration is acceptable when bounded and safe.

## Major Systems

### 1. Galaxy State Extraction
Responsible for:
- parsing save data
- extracting strategic information
- generating compact summaries

### 2. Historical Memory System
Responsible for:
- storing important historical events
- reputation persistence
- geopolitical history
- civilization memory

### 3. Personality System
Responsible for:
- stable personality traits
- behavior consistency
- strategic interpretation filters

### 4. Doctrine Planner
Responsible for:
- selecting strategic doctrines
- long-term strategic priorities
- geopolitical goals

### 5. LLM Client
Responsible for:
- local inference communication
- prompt management
- response validation

### 6. Mod Integration Layer
Responsible for:
- applying strategic weights safely
- communicating with Stellaris scripting systems
- graceful fallback behavior
- treating bridge values as optional hints, not authoritative commands

### 7. Strategic Request Worker
Responsible for:
- waiting idle until work is requested
- reading snapshot/request metadata
- running strategic analysis once per request
- writing decision JSON atomically enough for the mod to consume later
- avoiding realtime polling loops

### 8. File Bridge / Scripted Receiver
Responsible for:
- reading daemon response data in local development workflows
- validating a tiny set of bounded strategic values
- exposing only allowlisted scripted artifacts or receiver state
- hiding daemon/file failures from the scripted mod layer
- falling back to scripted defaults when unavailable

The bridge must not expose:
- raw prompts
- raw LLM output
- save data
- tactical orders
- fleet control
- economic micro-management

Initial bridge value contract:

```json
{
  "bridge": {
    "available": true,
    "doctrine": "balance_against_hegemon",
    "strategic_pressure": 0.74,
    "fallback_required": false
  }
}
```

The intended loop is:

```cpp
while (running)
{
    WaitForWork();
    ProcessStrategicRequest();
}
```

Not:

```cpp
while (true)
{
    ScanGameStateEveryFrame();
}
```

## Fail-Safe Architecture

If the daemon:
- crashes
- freezes
- times out
- produces invalid output

Then:
- the game continues normally
- vanilla AI continues functioning
- no save corruption occurs

## Initial MVP Scope

Version 0.1:
- save parsing
- strategic summary generation
- single doctrine selection
- JSON output
- basic AI strategic weight modification
- explicit request-driven analysis flow

No realtime control.
No tactical overrides.
No direct fleet management.
No synchronous gameplay blocking.

The first MVP may run as:

```text
1. save or snapshot exists
2. user/tool/bridge writes strategic_request JSON
3. daemon analyzes on request
4. daemon writes doctrine JSON
5. optional bridge exposes small values
6. mod uses values if available, otherwise scripted fallback
```

Current request-file MVP:

```text
Strategic Nexus.exe --request resources/strategic_request.example.json
```

Current sleepy daemon MVP:

```text
Strategic Nexus.exe --daemon exchange
```

Expected exchange directory:

```text
exchange/
    request.json
    response.json
    daemon.lock
    daemon.log
```

This exchange directory is not a vanilla mod API contract.
It is for the external daemon, optional bridge, and local debugging.

`daemon.lock` is a runtime heartbeat marker, not a permanent startup gate.
If the daemon crashes or the machine loses power, a leftover lock file must not block the next daemon start.
On startup, the daemon may overwrite a previous lock marker.
On clean shutdown, it removes its lock marker.

Responses are written through a temporary file and then published to `response.json`.
This prevents the mod from reading a half-written response during normal operation.

Responses include:
- `status`
- `request.request_id`
- `request.trigger`
- `decision` on success
- `bridge` bounded-value payload on success
- `error.message` on failure
