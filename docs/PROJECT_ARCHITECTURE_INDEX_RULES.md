# Strategic Nexus — Project Architecture Index Rules

## Core Rule

Strategic Nexus must maintain a centralized architectural index.

The project is becoming too large for:

* implicit assumptions
* scattered context
* undocumented subsystem relationships
* memory-only project navigation

The architecture must remain:

* navigable
* auditable
* maintainable
* modular
* contributor-friendly

---

# Design Philosophy

Strategic Nexus is evolving into a large-scale long-term architecture project.

The project already contains:

* daemon systems
* synchronization systems
* memory systems
* doctrine systems
* ethics systems
* faction systems
* probabilistic reasoning systems
* injection protection systems
* compatibility systems
* adaptive load systems
* multiplayer rules
* strategic persistence systems

Without structured indexing:

* context fragmentation will occur
* duplicated systems will appear
* architectural contradictions will accumulate
* onboarding difficulty will explode

A centralized architecture index is mandatory.

---

# Architecture Index Principle

The project should maintain:

* a global architectural map
* subsystem references
* dependency references
* implementation state tracking
* version tracking
* future roadmap visibility

The architecture index acts as:

* project memory
* developer navigation layer
* Codex orientation layer
* contributor onboarding layer

---

# Master Index Rule

The project should contain a master index document.

Suggested file:

```text id="l7k9xd"
docs/MASTER_ARCHITECTURE_INDEX.md
```

This document should reference:

* all major MD files
* subsystem categories
* implementation status
* dependency chains
* future planned systems

# Document Metadata Rule

Each major architecture document should eventually contain lightweight metadata.

Possible metadata:

* purpose
* subsystem category
* implementation priority
* dependency references
* implementation state
* schema impact
* runtime impact

This improves:

* maintainability
* automation
* tooling
* contributor understanding

---

# Implementation State Rule

Major systems should track implementation state.

Suggested states:

* conceptual
* planned
* prototype
* partially implemented
* implemented
* deprecated
* experimental
* disabled

This prevents:

* confusion
* duplicate work
* stale assumptions

---

# Dependency Tracking Rule

Subsystem relationships should be documented explicitly.

Example:

```text id="x5m7av"
Faction Governance
depends on:
- Ethics Systems
- Memory Systems
- Attention Systems
- Personality Systems
```

This prevents:

* circular architecture
* hidden coupling
* unsafe refactors

---

# Terminology Consistency Rule

Core terminology should remain standardized.

Examples:

* doctrine
* memory book
* active context
* strategic review
* payload
* daemon
* stale state
* host authority
* bounded reasoning

Avoid:

* multiple names for identical systems
* ambiguous subsystem terminology

---

# Architectural Authority Rule

The architecture index becomes the canonical navigation layer for the project.

Subsystem documents should not redefine:

* core philosophy
* authority hierarchy
* runtime ownership
* synchronization ownership

The index should clarify authoritative ownership.

---

# Branching And Conversation Continuity Rule

Strategic Nexus development may span:

* multiple chats
* multiple branches
* multiple developers
* long time periods

The architecture index should preserve continuity even when conversational context changes.

The project must not depend on:

* one continuous chat thread
* implicit remembered assumptions
* undocumented design history

---

# Codex Orientation Rule

The architecture index should help Codex:

* understand subsystem boundaries
* understand project philosophy
* avoid contradictory implementation
* discover dependencies
* understand priorities

Codex should not rely on fragmented historical chat context.

---

# Future Scaling Rule

The architecture index should remain scalable for:

* dozens of subsystems
* multiple contributors
* future tooling
* automatic validation
* future visualization tools
* long-term project evolution

The index itself becomes infrastructure.

---

# Design Goal

Strategic Nexus should behave like:

```text id="u2v8mk"
a structured long-term software architecture project with explicit subsystem organization
```

not:

```text id="n0x5cp"
a growing collection of disconnected markdown files and forgotten assumptions
```

Architectural organization is required for long-term project survival.
